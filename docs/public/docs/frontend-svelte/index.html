Svelte frontend

Operational guide for the alternate Svelte + Vite + Tauri workspace.

Current status

  • Svelte is the alternate desktop shell.
  • Radix/settings/transits parity with React is much better than before.
  • It includes a wired compute_transit_series flow through Tauri.
  • It still carries more compatibility-era assumptions than React, especially around storage/query helpers and in-memory fallbacks.
  • Performance and decomposition still need more care here than in the React shell.

Commands

npm install
npm run dev:svelte
npm run check:svelte
npm run build:svelte
npm run tauri:dev:svelte
npm run tauri:build:svelte
npm run docs:prepare

Workspace shape

  • apps/web-svelte/src/App.svelte — main integration shell.
  • apps/web-svelte/src/lib/components/ — feature-facing Svelte UI.
  • apps/web-svelte/src/lib/components/ui/ — shared Svelte primitive layer.
  • apps/web-svelte/src/lib/state/ — rune-backed application state.
  • apps/web-svelte/src/lib/stores/ — helper stores for icons, glyphs, data, and time navigation.
  • apps/web-svelte/src/lib/i18n/ — generated locale JSON plus runtime helper.

UI rules

  • Prefer shared Svelte UI primitives before bespoke controls.
  • Treat App.svelte as orchestration glue, not the ideal home for more feature logic.
  • Extract new feature UI into focused components instead of extending App.svelte with another large conditional branch.
  • Prefer shared tokens, variants, spacing, and wrappers over one-off component CSS.

Runtime behavior

  • Svelte can render from real chart payloads returned by Tauri.
  • It still uses some compatibility storage helpers and may fall back to in-memory results when those query paths return nothing.
  • The transit flow is real, but some UX and screen composition still lag behind React.
  • Recent fixes improved reactive stability, loading behavior, and repeated backend reload patterns.

Important current files

  • src/App.svelte — shell orchestration.
  • src/lib/components/MiddleContent.svelte — central content composition.
  • src/lib/components/RadixChart.svelte — wheel rendering.
  • src/lib/components/TimeNavigationPanel.svelte — time-navigation controls.
  • src/lib/state/layout.svelte.ts — workspace defaults and chart payload building.
  • src/lib/stores/timeNavigation.svelte.ts — shared time-navigation state.
  • src/lib/stores/data.svelte.ts — query helpers and compatibility fallbacks.

Parity summary

  • Mostly aligned:
    • translation pipeline
    • workspace defaults behavior
    • current-sky bootstrap
    • basic transits compute wiring
  • Still behind React:
    • app-shell decomposition
    • richer location and time input primitives
    • information view richness
    • some wheel polish and interaction detail
    • some storage/data-path cleanup

Recent stability improvements

  • language switching no longer bounces through redundant effect loops
  • wheel updates keep the last rendered state during loading
  • expensive recompute loops were reduced
  • repeated de440s.bsp reload behavior was reduced by backend-side caching
  • default flows no longer eagerly enable unsupported asteroid lookups

These paths are worth checking first before adding more reactive logic.

Shared assets

Svelte uses the same repo-root static/ source assets as React:

  • static/app-shell/
  • static/glyphs/
  • shared logos and favicon

Normalization happens in the frontend render layer, especially through:

  • apps/web-svelte/src/lib/stores/app-shell-icons.svelte.ts
  • apps/web-svelte/src/lib/stores/glyphs.svelte.ts
  • apps/web-svelte/src/lib/components/SharedSvgIcon.svelte

Legacy note:

  • apps/web-svelte/src/lib/icons/ should be treated as old local residue, not source of truth.

i18n

  • Source of truth: repo-root translations.csv.
  • Regeneration command: npm run i18n:sync.
  • Generated output includes apps/web-svelte/src/lib/i18n/*.json and the matching React locale files.
  • Author new copy in translations.csv, not in generated JSON.

Docs and Tauri integration

  • npm run docs:prepare builds Svelte alongside the other workspaces and publishes it under docs/static/apps/web-svelte/.
  • The base Tauri config still points at React by default.
  • Use:
    • src-tauri/tauri.react.conf.json
    • src-tauri/tauri.svelte.conf.json when switching foreground frontend targets.

Remaining priorities

  1. Decompose more logic out of App.svelte.
  2. Keep closing radix dashboard and wheel parity gaps with React.
  3. Port or redesign richer location/time input controls.
  4. Improve the information view and transits configurator UX.
  5. Reduce compatibility-era storage/query assumptions.
